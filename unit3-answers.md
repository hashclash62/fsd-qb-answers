## 1. Explain the purpose of the signature in a JWT token.

The signature in a JWT is used to verify that the token has not been changed after it was issued. A JWT usually contains user-related information such as username, user id, roles, and expiry time. Since the client stores and sends the token with every request, the server needs a way to confirm that the token is genuine.

The signature is created using the encoded header, encoded payload, a secret key or private key, and a signing algorithm such as HMAC SHA-256 or RSA.

```text
Signature = Algorithm(
    Base64Url(Header) + "." + Base64Url(Payload),
    Secret Key
)
```

When a request reaches the backend, the server recalculates the signature using the same secret key. If the recalculated signature matches the signature present in the token, the token is considered valid.

Purpose of the JWT signature:

- It proves that the token was created by a trusted server.
- It detects tampering in the header or payload.
- It helps prevent attackers from changing values such as role from `USER` to `ADMIN`.
- It supports stateless authentication because the server can validate the token without storing session data.

```text
Client sends JWT
      |
      v
Server recalculates signature
      |
      v
Is recalculated signature same as token signature?
      |
   +--+--+
   |     |
  Yes    No
   |     |
Valid  Reject token
```

## 2. Describe what happens when a JWT token is modified or tampered with.

When a JWT token is modified, its signature verification fails. This happens because the signature is calculated from the header and payload. If even one character in the header or payload changes, the signature generated by the server will no longer match the signature attached to the token.

For example, suppose a JWT payload contains:

```json
{
  "username": "rahul",
  "role": "USER"
}
```

If an attacker changes the role to:

```json
{
  "username": "rahul",
  "role": "ADMIN"
}
```

the old signature will not match the changed payload.

The server then treats the token as invalid and rejects the request. In a Spring Boot REST API, this usually results in an HTTP `401 Unauthorized` response.

```text
Original Token:
Header.Payload.Signature

Tampered Token:
Header.ModifiedPayload.OldSignature

Server validation:
Modified payload + secret key -> New signature

Old signature != New signature
Result: Token rejected
```

JWT tampering is detected only if the server properly verifies the signature. Therefore, secure JWT implementation must always validate the token before trusting any claim inside it.

## 3. Explain how JWT tokens help reduce server-side session storage.

Traditional session-based authentication stores session information on the server. After login, the server creates a session id and stores user details in server memory, database, or cache. The client stores only the session id in a cookie.

In JWT-based authentication, the token itself contains the required user information, also called claims. These claims may include username, user id, roles, and expiration time. Since the token is signed, the server can verify whether the token is valid without storing the session on the server.

```text
Traditional session authentication:

Client -> Session ID -> Server
                       |
                       v
                Session storage
                user details

JWT authentication:

Client -> JWT containing claims -> Server validates signature
                                  No server session required
```

This reduces server-side session storage because:

- The server does not need to store session data for every logged-in user.
- The token carries user identity and authorization claims.
- Any server instance can validate the token if it has the signing secret or public key.
- It works well in distributed and microservice-based systems.

However, JWT does not remove all server-side storage in every case. For example, applications may still use storage for refresh tokens, blacklisted tokens, or audit logs.

## 4. Explain what a JWT token is and why it is used in authentication.

JWT stands for JSON Web Token. It is a compact, URL-safe token format used to securely transfer information between a client and a server. In authentication, JWT is commonly used after successful login to identify the user in later requests.

After a user logs in with valid credentials, the backend generates a JWT and sends it to the client. The client stores the token, usually in memory, local storage, session storage, or an HTTP-only cookie. For future API requests, the client sends the token in the `Authorization` header.

```text
Authorization: Bearer <jwt-token>
```

Basic JWT authentication flow:

```text
User enters username/password
          |
          v
Spring Boot validates credentials
          |
          v
JWT is generated and returned
          |
          v
Client sends JWT with future requests
          |
          v
Server validates JWT and allows/denies access
```

JWT is used in authentication because:

- It supports stateless authentication.
- It avoids storing session information on the server.
- It can contain user roles and permissions.
- It is suitable for REST APIs and microservices.
- It can be verified using a digital signature.

In Spring Boot, JWT is commonly used with Spring Security filters. The filter extracts the token, validates it, reads the user details, and sets authentication in the Spring Security context.

## 5. Describe the structure of a JWT token: Header, Payload, Signature.

A JWT has three parts separated by dots:

```text
header.payload.signature
```

Example structure:

```text
xxxxx.yyyyy.zzzzz
  |     |     |
  |     |     +-- Signature
  |     +-------- Payload
  +-------------- Header
```

### Header

The header contains metadata about the token. It usually specifies the token type and signing algorithm.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### Payload

The payload contains claims. Claims are pieces of information about the user or token. Common claims include:

- `sub`: subject, usually username or user id
- `roles`: user roles
- `iat`: issued at time
- `exp`: expiration time

Example:

```json
{
  "sub": "student@example.com",
  "roles": ["USER"],
  "iat": 1710000000,
  "exp": 1710003600
}
```

The payload is Base64Url encoded, not encrypted. Therefore, sensitive data such as passwords should never be stored in JWT payload.

### Signature

The signature is created using the encoded header, encoded payload, secret key, and algorithm.

```text
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

The signature ensures integrity. If the token is modified, signature validation fails.

## 6. Explain what a JWT token is and why it is used in authentication.

A JWT, or JSON Web Token, is a digitally signed token used to represent authenticated user information. It is commonly used in modern Spring Boot REST APIs because REST APIs are usually stateless.

In JWT-based authentication, the user first logs in using valid credentials. After successful authentication, the server generates a JWT containing claims such as username, roles, and expiry time. The client sends this token with each protected API request.

```text
Login Request
 username + password
      |
      v
Authentication Manager
      |
      v
JWT Generator
      |
      v
JWT sent to client
      |
      v
Client uses JWT for protected APIs
```

JWT is used because:

- It allows the backend to identify users without maintaining server sessions.
- It helps secure REST endpoints.
- It can carry authorization data such as roles.
- It works well with mobile apps, web apps, and microservices.
- It improves scalability because multiple servers can validate the same token.

Important note: JWT is signed, not automatically encrypted. Anyone who has the token can decode the header and payload. Therefore, sensitive information like passwords, credit card numbers, or secrets should not be stored in the payload.

## 7. Explain how JUnit, Mockito, and MockMvc work together in backend testing.

JUnit, Mockito, and MockMvc are commonly used together to test Spring Boot backend applications.

JUnit is the testing framework. It provides annotations such as `@Test`, assertions such as `assertEquals`, and the structure for writing and running test cases.

Mockito is used to create mock objects. A mock object behaves like a real dependency but is controlled by the test. For example, while testing a service class, the repository can be mocked so that the test does not depend on the actual database.

MockMvc is used to test Spring MVC REST controllers without starting a full server. It can perform fake HTTP requests such as `GET`, `POST`, `PUT`, and `DELETE`, and verify the status code, response body, and headers.

```text
JUnit
 |
 |-- runs the test method
 |
 +-- Mockito
 |     |
 |     +-- mocks service/repository dependencies
 |
 +-- MockMvc
       |
       +-- sends fake HTTP requests to controller
       +-- checks response status and JSON output
```

Example use:

- JUnit defines the test case.
- Mockito mocks the service layer.
- MockMvc sends a request to the controller.
- Assertions verify that the response is correct.

Together, these tools allow backend testing at different levels: unit testing for services, isolated testing for dependencies, and controller testing for REST APIs.

## 8. Describe how mocking improves isolation and reliability of unit tests.

Mocking improves isolation by replacing real dependencies with controlled fake objects. In a Spring Boot application, a service class may depend on a repository, an external API, an email service, or another component. If a unit test uses all real dependencies, failures in those dependencies can make the unit test fail even when the unit being tested is correct.

With mocking, the test focuses only on the logic of the class being tested.

```text
Without mocking:

Service test -> Real Repository -> Real Database
                         |
                         v
                 Test fails if DB is down

With mocking:

Service test -> Mock Repository
                         |
                         v
                 Controlled test data
```

Benefits of mocking:

- Tests become independent of database availability.
- Tests run faster because they avoid external systems.
- Tests become predictable because mock responses are fixed.
- It is easier to test error conditions.
- It helps verify whether certain methods were called.

Example:

```java
when(userRepository.findById(1L)).thenReturn(Optional.of(user));
```

This tells Mockito to return a specific user when the repository method is called. The service test can then verify service behavior without connecting to a database.

Mocking improves reliability because test results depend only on the code under test and not on unstable external conditions.

## 9. Explain how JWT enables stateless authentication in Spring Boot applications.

JWT enables stateless authentication because the server does not need to store login session data for each user. Instead, after successful login, the server gives the client a signed token. This token contains claims such as username, roles, and expiry time.

For each protected request, the client sends the JWT in the request header:

```text
Authorization: Bearer <token>
```

In a Spring Boot application, a JWT filter usually performs the following steps:

```text
Incoming request
      |
      v
Extract JWT from Authorization header
      |
      v
Validate signature and expiry
      |
      v
Read username and roles from token
      |
      v
Set Authentication in SecurityContext
      |
      v
Controller method executes if authorized
```

The application becomes stateless because every request carries its own authentication information. The server validates each request independently.

Advantages:

- No server-side session storage is needed.
- APIs can scale across multiple server instances.
- It suits REST API architecture.
- It works well for distributed systems and microservices.

In Spring Security, stateless JWT authentication is often configured by disabling session creation:

```java
sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)
)
```

This tells Spring Security not to create or use HTTP sessions for authentication.

## 10. Describe how BCrypt password encoding works and why it is preferred over simple hashing.

BCrypt is a password hashing algorithm designed specifically for secure password storage. It does not store the original password. Instead, it converts the password into a hashed value using a salt and a computational cost factor.

BCrypt working:

```text
Plain password
      |
      v
Generate random salt
      |
      v
Apply BCrypt hashing with cost factor
      |
      v
Store BCrypt hash in database
```

During login, BCrypt does not decrypt the stored password. Hashing is one-way. Instead, it checks whether the entered password produces a matching hash using the salt and cost value stored inside the BCrypt hash.

```text
Login password + stored BCrypt hash
             |
             v
BCrypt matches?
             |
        +----+----+
        |         |
       Yes        No
        |         |
     Login     Reject
```

BCrypt is preferred over simple hashing because:

- It automatically uses a random salt, so identical passwords produce different hashes.
- It is intentionally slow, making brute-force attacks harder.
- It supports a cost factor, so security can be increased as hardware becomes faster.
- It protects against rainbow table attacks.

Simple hashing algorithms such as MD5 or SHA-256 are fast. Speed is useful for data integrity but dangerous for password storage because attackers can try millions of guesses per second. BCrypt reduces this risk by making each password guess computationally expensive.

In Spring Boot, BCrypt is commonly used through:

```java
BCryptPasswordEncoder passwordEncoder = new BCryptPasswordEncoder();
```

## 11. Explain the concept of role-based access control (RBAC) and how it is implemented in Spring Security.

Role-Based Access Control, or RBAC, is an authorization mechanism where access to resources is granted based on the user's role. Instead of assigning permissions individually to every user, users are assigned roles such as `USER`, `ADMIN`, or `MANAGER`. Each role has specific access rights.

Example:

```text
USER    -> view profile, view products
ADMIN   -> manage users, delete records, view reports
MANAGER -> approve requests, view team data
```

RBAC answers the question: "What is this authenticated user allowed to do?"

In Spring Security, RBAC can be implemented using URL-based rules or method-level annotations.

URL-based configuration:

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().authenticated()
);
```

Method-level security:

```java
@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/users/{id}")
public void deleteUser(@PathVariable Long id) {
    userService.deleteUser(id);
}
```

RBAC flow:

```text
Request with JWT
      |
      v
Authenticate user
      |
      v
Extract roles
      |
      v
Check required role for endpoint
      |
   +--+--+
   |     |
Allowed Denied
```

If a user is authenticated but does not have the required role, Spring Security returns `403 Forbidden`.

## 12. Describe the flow of authentication and authorization in Spring Security.

Authentication and authorization are two separate steps in Spring Security.

Authentication means verifying who the user is. Authorization means checking what the user is allowed to access.

Typical Spring Security flow:

```text
Client request
      |
      v
Security filter chain
      |
      v
Authentication filter checks credentials/token
      |
      v
AuthenticationManager validates user
      |
      v
UserDetailsService loads user details
      |
      v
PasswordEncoder or JWT validator verifies identity
      |
      v
SecurityContext stores Authentication object
      |
      v
Authorization rules check roles/permissions
      |
      v
Controller is accessed or request is rejected
```

For username/password login:

1. User submits username and password.
2. Spring Security passes credentials to `AuthenticationManager`.
3. `UserDetailsService` loads user details from database.
4. `PasswordEncoder` verifies the password.
5. If valid, the user is authenticated.
6. Authorization rules decide which endpoints the user can access.

For JWT-based APIs:

1. Client sends JWT in the `Authorization` header.
2. JWT filter extracts and validates the token.
3. User identity and roles are read from the token.
4. Authentication is placed in `SecurityContext`.
5. Spring Security checks endpoint access rules.

If authentication fails, the response is usually `401 Unauthorized`. If authentication succeeds but authorization fails, the response is `403 Forbidden`.

## 13. Explain how Spring Security protects REST APIs from unauthorized access.

Spring Security protects REST APIs using a filter chain that intercepts incoming requests before they reach controller methods. It checks whether the request is authenticated and whether the authenticated user has permission to access the requested endpoint.

For JWT-based REST APIs, protection usually works like this:

```text
HTTP Request
      |
      v
Spring Security Filter Chain
      |
      v
JWT Authentication Filter
      |
      v
Validate token
      |
      v
Check roles and permissions
      |
      v
Allow controller access or reject request
```

Spring Security protects APIs through:

- Authentication checks using login credentials or JWT tokens.
- Authorization rules using roles and permissions.
- Endpoint restrictions such as `/admin/**` requiring `ADMIN`.
- Password encoding using BCrypt.
- Protection from common attacks when configured properly.
- Exception handling for unauthorized and forbidden requests.

Example configuration:

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/auth/**").permitAll()
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .anyRequest().authenticated()
);
```

This means login endpoints are public, admin endpoints require the admin role, and all other endpoints require authentication.

If a request has no valid token, Spring Security returns `401 Unauthorized`. If the token is valid but the user lacks the required role, it returns `403 Forbidden`.

JUnit is a Java testing framework used to write and run automated test cases. In Spring Boot backend development, JUnit is used to test service methods, repository methods, utility classes, controllers, and other backend components.

- To verify that code works as expected.


```text

      |
```

- Arrange: prepare inputs and dependencies.

In backend testing, JUnit is often combined with Mockito for unit tests and MockMvc for controller tests.

## 15. Explain how Mockito helps in isolating components during unit testing.

Mockito helps isolate components by creating mock versions of dependencies. A class in a Spring Boot application often depends on other classes. For example, a service class may depend on a repository. If the test uses the real repository, it may require a real database. This makes the test slower and less reliable.

Mockito replaces the real dependency with a mock object.

```text
Class under test: UserService
Dependency: UserRepository

Test:
UserService -> Mock UserRepository
```

Example:

```java
@Mock
private UserRepository userRepository;

@InjectMocks
private UserService userService;

@Test
void shouldReturnUserById() {
    when(userRepository.findById(1L))
        .thenReturn(Optional.of(new User(1L, "Amit")));

    User user = userService.getUserById(1L);

    assertEquals("Amit", user.getName());
}
```

In this test, `UserService` is tested without a real database. Mockito controls what the repository returns.

Mockito helps in:

- Creating fake dependencies.
- Defining expected return values.
- Simulating exceptions.
- Verifying method calls.
- Testing only the logic of the target class.

Example verification:

```java
verify(userRepository).findById(1L);
```

This confirms that the repository method was called. By isolating components, Mockito makes tests faster, more focused, and easier to debug.

## 17. Explain the need for global exception handling and how @ControllerAdvice works.

Global exception handling is needed to provide consistent error responses across a Spring Boot REST API. Without it, different controllers may return different error formats for similar failures. This makes the API difficult to understand, test, and consume from frontend applications.

For example, one endpoint may return:

```json
{ "error": "User not found" }
```

Another may return:

```json
{ "message": "Record missing", "status": 404 }
```

This inconsistency can confuse frontend developers and users.

`@ControllerAdvice` is a Spring annotation used to define centralized exception handling logic. It can handle exceptions thrown by multiple controllers.

Example:

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(404, ex.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }
}
```

Flow:

```text
Controller throws exception
          |
          v
Spring searches @ControllerAdvice
          |
          v
Matching @ExceptionHandler is called
          |
          v
Standard error response is returned
```

Benefits:

- Consistent API error format.
- Cleaner controller code.
- Centralized exception management.
- Easier debugging and logging.
- Better frontend user experience.

Common exceptions handled globally include validation errors, resource not found errors, authentication errors, authorization errors, and internal server errors.

## 18. Describe the role of SLF4J and Logback in logging within Spring Boot applications.

SLF4J and Logback are used for logging in Spring Boot applications.

SLF4J stands for Simple Logging Facade for Java. It is not the actual logging implementation. It provides a common logging API. Developers write logging code using SLF4J interfaces.

Logback is the actual logging implementation used by default in Spring Boot. It decides how logs are printed, formatted, stored, and filtered.

```text
Application code
      |
      v
SLF4J API
      |
      v
Logback implementation
      |
      v
Console / File / External log system
```

Example:

```java
private static final Logger logger =
    LoggerFactory.getLogger(UserService.class);

logger.info("User created with id {}", userId);
logger.error("Failed to create user", exception);
```

Role of SLF4J:

- Provides a standard logging interface.
- Allows switching logging frameworks without changing application code.
- Supports parameterized logging.

Role of Logback:

- Writes logs to console or files.
- Supports log levels.
- Supports rolling log files.
- Allows configuration using `logback-spring.xml` or application properties.

Logging is important in Spring Boot because it helps developers understand application behavior, debug errors, monitor production systems, and track important events.

## 19. Explain different logging levels and their significance in debugging.

Logging levels define the importance or severity of log messages. They help developers control how much information is printed by the application.

Common logging levels:

### TRACE

Most detailed level. Used for very fine-grained information, such as step-by-step execution details.

### DEBUG

Used during development and debugging. It shows internal application details such as variable values, method calls, and decision points.

### INFO

Used for normal application events. Examples include application startup, successful user registration, or order creation.

### WARN

Indicates a potential problem that does not stop the application immediately. Example: slow API response or missing optional configuration.

### ERROR

Indicates a serious failure. Example: database connection failure, unhandled exception, or payment processing error.

```text
TRACE  -> Very detailed execution flow
DEBUG  -> Debugging information
INFO   -> Normal application events
WARN   -> Possible problem
ERROR  -> Serious failure
```

Significance in debugging:

- Developers can use `DEBUG` logs to understand why a bug occurs.
- `ERROR` logs help identify failures in production.
- `WARN` logs help detect problems before they become critical.
- Log levels reduce unnecessary output by filtering low-priority messages.

Example:

```properties
logging.level.com.example.demo=DEBUG
logging.level.org.springframework=INFO
```

In production, applications usually use `INFO`, `WARN`, and `ERROR`. `DEBUG` and `TRACE` are generally avoided in production unless troubleshooting is required because they can produce large logs and may expose sensitive details.

## 20. Describe how caching improves performance in Spring Boot applications.

Caching improves performance by storing frequently used data so that future requests can be served faster. Instead of repeatedly fetching the same data from a database or external API, the application can return the cached result.

Example without caching:

```text
Request 1 -> Service -> Database -> Response
Request 2 -> Service -> Database -> Response
Request 3 -> Service -> Database -> Response
```

Example with caching:

```text
Request 1 -> Service -> Database -> Store in cache -> Response
Request 2 -> Service -> Cache -> Response
Request 3 -> Service -> Cache -> Response
```

In Spring Boot, caching can be enabled using:

```java
@EnableCaching
```

A method can be cached using:

```java
@Cacheable("products")
public Product getProductById(Long id) {
    return productRepository.findById(id).orElseThrow();
}
```

Benefits of caching:

- Reduces database load.
- Improves API response time.
- Reduces repeated computation.
- Helps handle high traffic more efficiently.
- Improves scalability.

Common cache operations:

- `@Cacheable`: stores method result in cache.
- `@CachePut`: updates cache.
- `@CacheEvict`: removes stale data from cache.

Caching is useful for data that is read frequently but changes rarely, such as product categories, configuration values, and public reference data.

## 22. Describe the importance of testing Service and Repository layers separately.

Testing Service and Repository layers separately is important because both layers have different responsibilities.

The Service layer contains business logic. It decides what should happen in the application. For example, it may check rules before creating an order or calculate a discount.

The Repository layer handles database access. It performs operations such as saving, updating, deleting, and fetching data.

```text
Controller
   |
   v
Service layer       -> business logic
   |
   v
Repository layer    -> database access
   |
   v
Database
```

Why test them separately?

- It helps identify where a failure occurs.
- Service tests can run without a database by mocking repositories.
- Repository tests can focus only on database queries and mappings.
- Business logic and database logic are verified independently.
- Tests become easier to maintain.

Service layer test example:

```text
UserService test -> Mock UserRepository
```

Repository layer test example:

```text
UserRepository test -> Test database
```

In Spring Boot, service tests often use JUnit and Mockito. Repository tests often use `@DataJpaTest`, which loads only JPA-related components and can use an in-memory database like H2.

Separate testing improves reliability because each layer is checked according to its own responsibility.

## 23. Explain how dependency mocking improves test reliability.

Dependency mocking improves test reliability by removing dependence on external systems and unpredictable components. A unit test should test one unit of code. If it depends on a real database, network service, file system, or third-party API, the test may fail for reasons unrelated to the code being tested.

Mocking replaces these dependencies with controlled objects.

```text
Unreliable unit test:

Test -> Service -> Real API / Database
                  |
                  v
             May fail due to external issue

Reliable unit test:

Test -> Service -> Mock API / Mock Repository
                  |
                  v
             Predictable response
```

Dependency mocking improves reliability because:

- Tests do not fail due to database downtime.
- Tests do not depend on network availability.
- Test data is controlled.
- Error scenarios can be simulated easily.
- Tests run faster and produce consistent results.

Example:

```java
when(paymentGateway.charge(any()))
    .thenThrow(new PaymentFailedException("Payment failed"));
```

This allows the service error-handling logic to be tested without calling a real payment gateway.

Mocking also helps verify interactions:

```java
verify(emailService).sendWelcomeEmail(user);
```

This confirms that the expected dependency method was called. As a result, tests become focused, repeatable, and easier to debug.

## 24. Describe how authentication errors and exceptions are handled in REST APIs.

Authentication errors occur when the user identity cannot be verified. In REST APIs, this may happen because of missing credentials, invalid username/password, expired JWT, invalid JWT signature, or malformed token.

Common authentication errors:

- Missing token
- Invalid token
- Expired token
- Wrong username or password
- Disabled account

Typical HTTP responses:

- `401 Unauthorized`: user is not authenticated.
- `403 Forbidden`: user is authenticated but not allowed to access the resource.

Flow:

```text
Request to protected API
      |
      v
Spring Security checks authentication
      |
      v
Token missing/invalid/expired?
      |
      v
Return 401 Unauthorized
```

In Spring Security, authentication exceptions can be handled using:

- `AuthenticationEntryPoint` for unauthenticated requests.
- `AccessDeniedHandler` for unauthorized access after authentication.
- `@ControllerAdvice` for application-level exceptions.

Example error response:

```json
{
  "status": 401,
  "error": "Unauthorized",
  "message": "Invalid or expired token",
  "path": "/api/orders"
}
```

Good REST API error handling should:

- Return correct HTTP status codes.
- Use a consistent error response format.
- Avoid exposing sensitive security details.
- Log the error for debugging.
- Give enough information for the client to respond properly.

For example, the API should say "Invalid credentials" instead of revealing whether the username or password was wrong separately.

## 25. Explain the difference between authentication and authorization with examples.

Authentication and authorization are related but different concepts.

Authentication means verifying the identity of the user. It answers: "Who are you?"

Authorization means checking what the authenticated user is allowed to do. It answers: "What are you allowed to access?"

```text
Authentication:
User proves identity
Example: login with username and password

Authorization:
System checks permission
Example: only ADMIN can delete users
```

Example scenario:

1. A student logs into a college portal using email and password.
2. The system verifies the credentials. This is authentication.
3. The student tries to open the admin dashboard.
4. The system checks the student's role.
5. Since the student does not have the `ADMIN` role, access is denied. This is authorization.

Comparison:

| Point | Authentication | Authorization |
|---|---|---|
| Meaning | Verifies identity | Verifies access rights |
| Question | Who are you? | What can you access? |
| Happens | First | After authentication |
| Example | Login using password | Checking `ADMIN` role |
| Failure status | 401 Unauthorized | 403 Forbidden |

In Spring Security, authentication may be handled using username/password or JWT validation. Authorization may be handled using roles, permissions, URL rules, or annotations such as `@PreAuthorize`.

## 26. Describe how token expiration works in JWT-based systems.

JWT token expiration is controlled using the `exp` claim. The `exp` claim stores the time after which the token is no longer valid. When the server receives a JWT, it checks both the signature and the expiration time.

```json
{
  "sub": "student@example.com",
  "role": "USER",
  "exp": 1710003600
}
```

JWT expiration flow:

```text
Client sends JWT
      |
      v
Server validates signature
      |
      v
Server checks exp claim
      |
      v
Is current time before expiry?
      |
   +--+--+
   |     |
  Yes    No
   |     |
Allow  Reject with 401
```

Token expiration is important because JWTs are usually stateless. If a token is stolen, the attacker may use it until it expires. A shorter expiry time reduces this risk.

Common approach:

- Access token: short lifespan, such as 15 minutes or 1 hour.
- Refresh token: longer lifespan, used to get a new access token.

When an access token expires, the client must either:

- Ask the user to log in again, or
- Use a valid refresh token to request a new access token.

In Spring Boot, JWT libraries usually throw an exception when an expired token is parsed. The application should handle this and return a clear `401 Unauthorized` response.

## 28. A user logs into an application and receives a JWT token but is denied access to certain endpoints. Explain how authentication and authorization work in this scenario and why access is restricted.

In this scenario, the user has successfully completed authentication but has failed authorization for certain endpoints.

Authentication happens during login. The user provides valid credentials such as username and password. Spring Security verifies these credentials using `AuthenticationManager`, `UserDetailsService`, and `PasswordEncoder`. If the credentials are correct, the server generates a JWT token and sends it to the user.

The JWT proves that the user is authenticated.

```text
Login
 username/password
      |
      v
Credentials valid
      |
      v
JWT issued
      |
      v
User is authenticated
```

When the user calls another endpoint, the JWT is sent in the request:

```text
Authorization: Bearer <jwt-token>
```

Spring Security validates the token. If the token is valid, the user identity and roles are loaded into the security context.

Authorization then checks whether the user's role is allowed for the endpoint.

```text
Request to /admin/users
      |
      v
JWT valid?
      |
      v
User role = USER
      |
      v
Endpoint requires ADMIN
      |
      v
Access denied: 403 Forbidden
```

Access is restricted because authentication alone does not give permission to every resource. A logged-in user may have the `USER` role, but an endpoint such as `/admin/**` may require the `ADMIN` role.

Possible reasons for denial:

- The user does not have the required role.
- The JWT does not contain the expected role claim.
- Spring Security configuration requires a different role.
- There is a mismatch between `ROLE_ADMIN` and `ADMIN`.
- Method-level security such as `@PreAuthorize` blocks access.

Correct distinction:

- If the token is invalid or missing, response should be `401 Unauthorized`.
- If the token is valid but role is insufficient, response should be `403 Forbidden`.

Thus, the user is authenticated but not authorized to access restricted endpoints.

## 29. A system uses BCrypt for password storage, but login response time is slow. Explain how BCrypt works and why it affects performance.

BCrypt is designed to be slow. This is not a bug; it is a security feature. BCrypt protects passwords by making brute-force password guessing expensive.

When a password is stored, BCrypt performs the following steps:

```text
Plain password
      |
      v
Generate random salt
      |
      v
Apply BCrypt algorithm repeatedly
based on cost factor
      |
      v
Store hash in database
```

The stored BCrypt value contains:

- Algorithm version
- Cost factor
- Salt
- Password hash

During login, the entered password is checked against the stored hash. BCrypt must repeat the hashing process using the stored salt and cost factor.

```text
User enters password
      |
      v
Fetch stored BCrypt hash
      |
      v
Run BCrypt match operation
      |
      v
Password matches?
```

BCrypt affects performance because of the cost factor. The cost factor controls how many rounds of computation are performed. A higher cost factor means stronger protection but slower password verification.

Example:

```text
Low cost  -> faster login, weaker brute-force resistance
High cost -> slower login, stronger brute-force resistance
```

Why this slowness is useful:

- Attackers cannot test millions of passwords quickly.
- Stolen password hashes are harder to crack.
- It reduces the effectiveness of brute-force and dictionary attacks.

However, if login response time is too slow, possible improvements include:

- Choose a balanced BCrypt strength value.
- Avoid unnecessary database or API calls during login.
- Use proper indexing on username/email columns.
- Monitor login endpoint performance.
- Do not replace BCrypt with fast hashing like MD5 or SHA-256 for passwords.

The goal is to keep login reasonably fast for real users but slow enough to protect against attackers.

## 30. An API exposes admin-level data to all users due to misconfiguration. Explain how role-based access control (RBAC) should function and what might have gone wrong.

RBAC should ensure that users can access only the endpoints allowed for their roles. Admin-level data should be accessible only to users with the `ADMIN` role or equivalent permission.

Expected RBAC behavior:

```text
Request to /admin/reports
      |
      v
Is user authenticated?
      |
      v
Does user have ADMIN role?
      |
   +--+--+
   |     |
  Yes    No
   |     |
Allow  Return 403 Forbidden
```

In Spring Security, admin endpoints should be configured like this:

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().authenticated()
);
```

Or at method level:

```java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin/reports")
public List<Report> getReports() {
    return reportService.getReports();
}
```

What might have gone wrong:

- Admin endpoints were configured with `permitAll()`.
- Rule order was incorrect, and a general rule allowed all requests before admin rules.
- The endpoint path did not match the configured pattern.
- Method-level security was not enabled.
- The code used `hasRole("ADMIN")` but authorities were stored incorrectly.
- The application confused `ROLE_ADMIN` and `ADMIN`.
- JWT did not include correct role claims.
- Controller did not have security annotations.
- Tests did not check access with different roles.

Example of dangerous misconfiguration:

```java
.requestMatchers("/admin/**").permitAll()
```

Another common issue:

```java
.anyRequest().permitAll()
```

This allows every endpoint unless stricter rules are applied earlier.

To fix the issue, the application should enforce strict RBAC rules, verify JWT roles, test endpoints using MockMvc with different roles, and return `403 Forbidden` for authenticated users without admin permission.

## 32. During testing, unit tests fail when the database is unavailable. Explain how Mockito helps in isolating dependencies and improving test reliability.

If unit tests fail when the database is unavailable, those tests are not properly isolated. A unit test should test a small unit of logic, such as a service method, without depending on external systems like databases.

Mockito solves this by replacing the real repository with a mock repository.

Problem:

```text
Unit test -> Service -> Real Repository -> Database
                                      |
                                      v
                              DB unavailable
                                      |
                                      v
                                Test fails
```

Better test with Mockito:

```text
Unit test -> Service -> Mock Repository
                                      |
                                      v
                              Fixed test response
```

Example:

```java
@Mock
private UserRepository userRepository;

@InjectMocks
private UserService userService;

@Test
void shouldReturnUserName() {
    User user = new User(1L, "Amit");

    when(userRepository.findById(1L))
        .thenReturn(Optional.of(user));

    String name = userService.getUserName(1L);

    assertEquals("Amit", name);
}
```

In this test, the database is not required. The repository response is controlled by Mockito.

How Mockito improves reliability:

- Tests pass or fail based on service logic, not database availability.
- Tests run faster.
- Tests are predictable and repeatable.
- Error scenarios can be simulated easily.
- Developers can verify interactions with dependencies.

Example of testing failure case:

```java
when(userRepository.findById(99L))
    .thenReturn(Optional.empty());
```

This allows the service's "user not found" logic to be tested without creating database records.

Important distinction:

- Service unit tests should mock repositories.
- Repository integration tests may use a test database with `@DataJpaTest`.

Thus, Mockito improves test reliability by isolating the unit under test from unavailable or unstable dependencies.

## 33. A system slows down due to repeated API calls for the same data. Explain how caching improves performance and its possible risks.

Caching improves performance by storing frequently requested data and reusing it for future requests. If the same data is requested repeatedly, the application can return it from cache instead of repeatedly calling the database or another API.

Problem without caching:

```text
Client request -> Backend -> Database/API -> Response
Client request -> Backend -> Database/API -> Response
Client request -> Backend -> Database/API -> Response
```

With caching:

```text
First request -> Backend -> Database/API -> Cache -> Response
Next request  -> Backend -> Cache -> Response
```

In Spring Boot, caching can be implemented using annotations:

```java
@Cacheable("products")
public Product getProductById(Long id) {
    return productRepository.findById(id).orElseThrow();
}
```

When this method is called for the first time, the result is fetched from the database and stored in the cache. Later calls with the same input return the cached value.

Benefits:

- Faster API response time.
- Reduced database load.
- Reduced external API calls.
- Better scalability under high traffic.
- Lower repeated computation.

Possible risks of caching:

- Stale data: cache may return old data after the database is updated.
- Memory usage: large caches can consume too much memory.
- Cache invalidation complexity: deciding when to remove or update cached data can be difficult.
- Security risk: user-specific sensitive data may be cached incorrectly.
- Inconsistent data: multiple application instances may have different cache values.
- Cache stampede: many requests may hit the database at once when a cache entry expires.

To reduce risks:

- Use `@CacheEvict` when data changes.
- Set proper time-to-live values.
- Avoid caching sensitive user-specific data unless carefully scoped.
- Choose suitable cache providers such as Caffeine, Redis, or Ehcache.
- Monitor cache hit rate and memory usage.

Caching is most useful for data that is read frequently and changes rarely. It should be used carefully so that performance improves without returning incorrect or outdated data.
